# Stage 18.4A.1 — Redirect Compatibility Report

## Issue

Stage 18.4A migrated the Filament administration portal from:

```
https://api.vestradetergents.com/admin
```

to a dedicated subdomain:

```
https://admin.vestradetergents.com
```

Filament was reconfigured with:

```php
->domain(config('app.admin_domain'))
->path('')
```

The original nginx redirect preserved the `/admin` path segment, so a legacy bookmark such as `https://api.vestradetergents.com/admin/login` redirected to `https://admin.vestradetergents.com/admin/login`, which returned a 404 because Filament no longer serves `/admin/*`.

## Redirect Mappings

| Old URL (api.vestradetergents.com) | New URL (admin.vestradetergents.com) |
|------------------------------------|--------------------------------------|
| `/admin` | `/` |
| `/admin/` | `/` |
| `/admin/login` | `/login` |
| `/admin/dashboard` | `/dashboard` |
| `/admin/orders` | `/orders` |
| `/admin/products` | `/products` |
| `/admin/customers` | `/customers` |
| `/admin/reports` | `/reports` |
| `/admin/settings` | `/settings` |
| `/admin/orders?tableSearch=123` | `/orders?tableSearch=123` |

## Implementation

File: `nginx/conf.d/vestra.conf.template`

Replaced the two literal `/admin` redirect locations with a single regex location inside the API vhost:

```nginx
# Filament admin has moved to ${ADMIN_DOMAIN}. Redirect old /admin URLs
# without preserving the /admin prefix, because Filament now uses ->path('').
location ~ ^/admin(/.*)?$ {
    return 301 https://${ADMIN_DOMAIN}$1$is_args$args;
}
```

### How it works

- `^/admin(/.*)?$` matches `/admin` and `/admin/anything/here`.
- The optional capture group `(/.*)?` captures the remainder of the path (empty for `/admin`).
- `$1` is the captured remainder, preserving deep paths such as `/admin/orders/123/edit`.
- `$is_args$args` preserves the query string, including table filters and search terms.
- The redirect is a permanent HTTP 301, safe for search engines and bookmarks.

## Local Validation

The template was rendered using the same `envsubst` logic the `nginx:1.27-alpine` entrypoint uses, with mock certificates and upstream hosts. The resulting configuration passed `nginx -t` and the redirect targets were verified with `curl`.

### Rendered redirect block

```nginx
# Filament admin has moved to admin.vestradetergents.com. Redirect old /admin URLs
# without preserving the /admin prefix, because Filament now uses ->path('').
location ~ ^/admin(/.*)?$ {
    return 301 https://admin.vestradetergents.com$1$is_args$args;
}
```

### Verified redirect targets

| Request | Response `Location:` header |
|---------|-----------------------------|
| `https://api.vestradetergents.com/admin` | `https://admin.vestradetergents.com` |
| `https://api.vestradetergents.com/admin/` | `https://admin.vestradetergents.com/` |
| `https://api.vestradetergents.com/admin/login` | `https://admin.vestradetergents.com/login` |
| `https://api.vestradetergents.com/admin/orders` | `https://admin.vestradetergents.com/orders` |
| `https://api.vestradetergents.com/admin/products` | `https://admin.vestradetergents.com/products` |
| `https://api.vestradetergents.com/admin/customers` | `https://admin.vestradetergents.com/customers` |
| `https://api.vestradetergents.com/admin/orders?tableSearch=123` | `https://admin.vestradetergents.com/orders?tableSearch=123` |

All responses returned `HTTP/2 301`. No redirect loops are possible because the regex only matches paths beginning with `/admin` on the API domain, and the target domain is different.

## Production Validation Commands

Run from the VPS after deployment:

```bash
curl -sI https://api.vestradetergents.com/admin
curl -sI https://api.vestradetergents.com/admin/login
curl -sI https://api.vestradetergents.com/admin/orders
curl -sI https://api.vestradetergents.com/admin/products
curl -sI https://api.vestradetergents.com/admin/customers
```

Expected for each: `HTTP/2 301` with a `location:` header pointing to the equivalent path on `https://admin.vestradetergents.com` and no `/admin` segment in the target.
