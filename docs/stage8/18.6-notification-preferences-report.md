# Stage 18.6 — Notification & Account Preferences Report

## Objective

Allow customers to control notification preferences and account-level preferences.

## Implementation

### Backend

- Migration `create_customer_preferences_table` added:
  - `user_id`, `notification_preferences` (json), `account_preferences` (json).
- `CustomerPreference` model created.
- New `PreferenceController` handles GET/PUT `/auth/preferences`.
- `UpdatePreferenceRequest` validates preferences as arrays.
- `PreferenceResource` returns preferences with empty objects instead of null.

### Frontend

- New `/account/preferences` page with toggles for:
  - Email notifications
  - SMS notifications
  - Order updates
  - Marketing emails
  - Promotions
  - Newsletter
  - Preferred language
  - Preferred currency
  - Communication preferences
- Preferences save immediately with toast feedback.

### API Endpoints

| Method | Endpoint | Purpose |
|--------|----------|---------|
| GET | `/api/v1/auth/preferences` | Get preferences |
| PUT | `/api/v1/auth/preferences` | Update preferences |

## Validation

- Preferences must be arrays.
- Each preference row is unique per user.
- Auto-created on first fetch if missing.
