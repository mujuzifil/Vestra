# Stage 18.7 — Company Profile Report

## Objective

Provide approved distributors with a self-service company profile where they can manage corporate information, branches, contacts, documents, and branding.

## Distributor Model & Profile Fields

The `App\Models\Distributor` model (`backend/app/Models/Distributor.php`) stores the master distributor record. Key fields include:

- **Company identity:** `company_name`, `trading_name`, `registration_number`, `tax_identification`, `vat_number`, `business_type`, `industry`, `years_in_business`, `company_size`, `website`
- **Contact:** `primary_contact_name`, `email`, `phone`
- **Address:** `country`, `district`, `city`, `address`, `postal_address`
- **Branding & settings:** `logo_path`, `operating_hours_json`, `bank_info_json`, `billing_info_json`
- **Commercial:** `expected_monthly_volume`, `products_of_interest`
- **Status:** `status` (`active` or `suspended`), `approved_at`, `suspended_at`

The model also defines relationships to `User`, `DistributorBranch`, `DistributorContact`, `DistributorDocument`, `Order`, `QuotationRequest`, `PaymentUpload`, `CreditAccount`, and `DistributorProductPrice`. Convenience methods include `defaultBranch()`, `primaryContact()`, `isActive()`, and `logoUrl()`.

## Branch Management

Distributors can maintain multiple delivery or billing branches. The `DistributorBranch` model supports name, manager, contact details, address, geolocation, delivery notes, status, and a single `is_default` flag. The model's `setAsDefault()` method ensures only one branch is default per distributor. See `18.7-branch-management-report.md` for the full branch API and frontend details.

## Contact Persons

Contacts are stored in `distributor_contacts`. Each contact has a name, role, phone, email, an `is_primary` flag, and a `permissions_json` array. Like branches, only one contact can be primary at a time. The portal exposes full CRUD operations under `/api/v1/distributor/contacts`.

## Documents

Distributors can upload certificates, tax documents, and other compliance files via `/api/v1/distributor/documents`. Documents store a title, type, file path, uploader, and version. The `StoreDocumentRequest` enforces a 10 MB file limit and validates the title. Uploaded files are stored on the `public` disk under `distributor-documents/` and are scoped to the owning distributor.

## Logo Upload & Audit Logging

The company profile page allows logo upload and removal:

- `POST /api/v1/distributor/profile/logo` accepts an image file, replaces any existing logo on the `public` disk, and updates `logo_path`.
- `DELETE /api/v1/distributor/profile/logo` removes the stored logo and clears the path.

Both actions, as well as profile updates, are logged via `AuditService` with the user's IP address and user agent. The `DistributorProfileResource` returns the public `logo_url` and the associated `credit_account` when loaded.

## API Endpoints

| Method | Endpoint | Purpose |
|--------|----------|---------|
| GET | `/api/v1/distributor/profile` | View company profile |
| PUT | `/api/v1/distributor/profile` | Update profile fields |
| POST | `/api/v1/distributor/profile/logo` | Upload company logo |
| DELETE | `/api/v1/distributor/profile/logo` | Remove company logo |
| GET/POST/PUT/DELETE | `/api/v1/distributor/branches/*` | Branch CRUD |
| GET/POST/PUT/DELETE | `/api/v1/distributor/contacts/*` | Contact CRUD |
| GET/POST/DELETE | `/api/v1/distributor/documents/*` | Document list/upload/delete |

All endpoints require `auth:sanctum` and the `distributor` middleware. Updates are authorized through `DistributorPolicy`, which restricts mutation to the owning user or an administrator.
