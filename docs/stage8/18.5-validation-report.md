# Stage 18.5 — Validation Report

## Scenarios Covered

| Scenario | Validation |
|----------|------------|
| Guest orders | Guest checkout auto-registers user; subsequent order access requires authentication. |
| Registered orders | `OrderController::show()` scopes orders to the authenticated user. |
| Paid orders | Payment status badge renders correctly; timeline shows payment event; invoice download available. |
| COD orders | Skip payment retry button; stock decremented on order creation. |
| Cancelled orders | Timeline hidden; cancellation banner shown; payment retry disabled. |
| Refunded orders | Timeline hidden; refund banner shown; payment retry disabled. |
| Multiple orders | Orders list filtered/sorted/paginated client-side for responsiveness. |
| Large orders | Invoice PDF and order detail render all items without arbitrary limits. |
| Partial fulfilment | Tracking section shows placeholders until courier and tracking number are set by admin. |

## Security

- Invoice download verifies order ownership (`user_id === auth user`).
- Tracking page only searches orders belonging to the authenticated user.
- Order detail page requires authentication via `useAuth`.
- `OrderResource` admin access restricted to admins.

## Performance

- Orders are loaded with `items` relationship in a single query per page.
- Timeline is computed on the backend and returned with the order resource.
- Filtering and pagination are client-side to avoid repeated API calls.

## Known Limitations

- No server-side pagination on the orders list; very large order histories will load all records.
- Guest tracking is not implemented; tracking requires login.
- Courier tracking number lookup is a placeholder pending logistics integration.
