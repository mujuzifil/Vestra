# Stage 18.7 — Distributor Registration Report

## Objective

Enable prospective partners to apply for a VESTRA distributor account through a public form, track their application status, and be onboarded automatically once an administrator approves the request.

## Public Application Form

The distributor application is exposed at `/distributor`. The page is rendered by `frontend/app/distributor/distributor-page-client.tsx` and presents a marketing landing experience with benefits, FAQs, and the application form powered by `frontend/components/forms/distributor-form.tsx`.

The form collects:

- Full name
- Business name
- Email address
- Phone number
- City / region
- Business type (retailer, wholesaler, commercial laundry, other)
- Business experience / target market description

Client-side validation enforces required fields, email format, and a minimum experience length. On submit, the `useDistributorMutation` hook calls `POST /api/v1/distributor`, mapping the frontend camelCase fields to the backend snake_case payload.

The backend controller `App\Http\Controllers\Api\V1\DistributorController::store` uses `StoreDistributorRequest` to validate:

- `company_name` (required, string, max 255)
- `contact_person` (required, string, max 255)
- `email` (required, valid email)
- `phone`, `address`, and `business_description` (optional)

The route is protected by the `throttle:distributor` middleware to prevent abuse.

## Review Workflow

Applications are stored in `distributor_requests`. The `DistributorStatus` enum drives the workflow:

| Status | Meaning |
|--------|---------|
| `pending` | Newly submitted, awaiting admin attention |
| `under_review` | Being evaluated by the partnerships team |
| `information_requested` | Additional documentation or clarification required |
| `approved` | Application accepted; distributor account created |
| `rejected` | Application declined |

Administrators review requests through the Filament `DistributorRequestResource` (`backend/app/Filament/Resources/DistributorRequestResource.php`). The table supports searching, status/priority filters, bulk approve/reject/request-information actions, and assignment tracking.

## Admin Approval & Onboarding

When an admin clicks **Approve**, `DistributorOnboardingService::approve` runs the following inside a database transaction:

1. Sets the request status to `approved`.
2. Resolves or creates a `User` record for the applicant email.
3. Creates the `Distributor` record seeded from the request data (company name, contact, address, etc.).
4. Seeds a default `Head Office` branch as the default delivery location.
5. Seeds a default primary contact person.
6. Creates a `CreditAccount` with `limit = 0`, `balance = 0`, `authorized_amount = 0`, and `status = pending`.
7. Assigns the `distributor` Spatie role to the user.
8. Sends the `DistributorApprovedNotification`.
9. Logs the action via `AuditService`.

Rejection updates the request status to `rejected` and records an audit entry with the reason. Once rejected, the record cannot be approved without first returning it to review.

## Application Status Page

Authenticated users can view the status of their most recent application at `GET /api/v1/distributor/application-status`. The endpoint returns the latest `DistributorRequest` matching the user's email, including company name, contact information, status, and timestamps. The frontend uses this on the account dashboard to inform applicants whether their submission is pending, under review, approved, or rejected.

## Security & Audit

- Public form rate-limited via `throttle:distributor`.
- Approval actions in Filament require confirmation to prevent accidental onboarding.
- Every approval, rejection, and status change is written to the audit log with the acting admin, IP address, and user agent.
