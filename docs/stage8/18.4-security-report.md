# Stage 18.4 — Security Report

## Webhook Signature Verification

The webhook endpoint `POST /api/v1/payments/callback` is protected by `PaymentCallbackRequest`:

- Reads the `verif-hash` request header.
- Computes the HMAC-SHA256 of the raw payload using the webhook secret.
- Falls back to the Flutterwave secret key if no webhook secret is configured.
- Uses `hash_equals()` to prevent timing attacks.
- Rejects the request with 403 if the signature is missing or invalid.
- Logs failed attempts with IP and user agent.

## Callback Handling

- Customer redirect callbacks are handled by the frontend return page.
- The frontend immediately calls the authenticated `/payments/{reference}/verify` endpoint.
- The backend re-verifies the transaction state directly with Flutterwave; the redirect `status` parameter is used only for UI messaging.

## Server-Side Validation

| Validation | Implementation |
|------------|----------------|
| Amount match | `PaymentService::amountMatches()` compares Flutterwave amount/currency with the order total |
| Currency | Hard-coded to `UGX`; rejects mismatched currencies |
| Transaction ownership | `getTransaction()` scopes queries to the authenticated user's orders |
| Order ownership | `PaymentController::initiate()` verifies the order belongs to the authenticated user |
| Idempotency | `verifyAndProcess()` skips already-successful transactions |
| Stock decrement | Guarded by `stock_decremented` flag on the order |

## Secret Management

- No Flutterwave keys are hard-coded in source.
- Keys are read from environment variables through `config/services.php`.
- The secret key is never exposed to the frontend.
- The encryption key is reserved for future use.

## CSRF / Session

- API routes are stateless and use Sanctum tokens.
- No session-based CSRF concerns for payment endpoints.

## Recommended Hardening

1. Set a dedicated `FLUTTERWAVE_WEBHOOK_SECRET` in production rather than relying on the fallback to `FLUTTERWAVE_SECRET_KEY`.
2. Restrict the webhook route to Flutterwave IP ranges at the nginx or application firewall level if those ranges are documented.
3. Add monitoring and alerting on webhook signature failures.
4. Implement webhook replay idempotency through the existing `success` status guard.
