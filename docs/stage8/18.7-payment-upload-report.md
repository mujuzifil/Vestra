# Stage 18.7 — Payment Upload Report

## Objective

Allow distributors to upload proof of bank transfer payments so the finance team can verify them against outstanding invoices.

## Upload Flow

Distributors submit payment proof at `POST /api/v1/distributor/payments`. The request is validated by `StorePaymentUploadRequest`:

- `amount` — required, numeric, minimum 0.01, maximum 999,999,999.99
- `reference_number` — required, string, max 255
- `file` — required, `mimes:png,jpg,jpeg,pdf`, max 5 MB
- `notes` — optional, max 1000 characters

`PaymentUploadController::store` saves the file to the `public` disk under `payment-uploads/`, creates a `PaymentUpload` record linked to the distributor, sets the status to `uploaded`, and logs the action via `AuditService`. The file hash name prevents collisions and avoids exposing original filenames directly in storage paths.

## Status Workflow

The `PaymentUploadStatus` enum defines the lifecycle:

| Status | Meaning |
|--------|---------|
| `uploaded` | Just submitted by the distributor |
| `pending_verification` | Queued for finance review |
| `verified` | Approved and applied against the account |
| `rejected` | Declared invalid or unmatched |

The initial upload sets the status to `uploaded`. A background or admin verification process can transition the record to `pending_verification`, `verified`, or `rejected`, populating `verified_by`, `verified_at`, and `verification_notes`. Once verified, the corresponding payment can be applied to the distributor's credit account as a `payment` transaction, reducing the outstanding balance.

## Storage & Audit

Files are stored on the public disk and accessible through a signed or public URL returned as `file_url`. The upload event is audited with the amount and reference number. The list endpoint `GET /api/v1/distributor/payments` returns the distributor's own uploads in descending chronological order.

## Frontend

The payments page at `/distributor/payments` (`frontend/app/distributor/payments/payments-page-client.tsx`) provides:

- A form for amount, reference number, notes, and receipt file upload.
- A drag-and-click file target with accepted formats (PDF, JPG, JPEG, PNG).
- A history table showing reference, amount, status badge, date, and a link to view the receipt.
- Responsive `overflow-x-auto` wrapping for mobile.

The frontend performs basic client-side checks for amount and reference number before submission, reducing unnecessary API calls. Status badges use color coding matching the backend enum so distributors can immediately distinguish verified, pending, and rejected uploads.
