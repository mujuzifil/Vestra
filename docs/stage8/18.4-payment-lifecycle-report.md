# Stage 18.4 — Payment Lifecycle Report

## Payment Transaction States

| State | Meaning | Transition Trigger |
|-------|---------|-------------------|
| `initiated` | Transaction record created, Flutterwave request in progress | `PaymentService::initiate()` |
| `pending` | Flutterwave link generated, awaiting customer payment | Successful `/payments` response |
| `success` | Payment confirmed by Flutterwave | `verifyAndProcess()` confirms successful verification |
| `failed` | Initiation failed or verification returned failure | Flutterwave error or verification mismatch |

## Order Status Lifecycle

```
Pending
  ↓ Payment confirmed
Paid
  ↓ Admin action
Processing
  ↓ Admin action
Packed
  ↓ Admin action
Shipped
  ↓ Admin action
Delivered
```

Alternative states:

```
Pending → Cancelled (admin or customer)
Paid → Refunded (admin)
Processing → Cancelled (admin)
Packed → Cancelled (admin)
```

## Payment Status Lifecycle

```
Pending
  ↓ verifyAndProcess succeeds
Paid
  ↓ refund
Refunded
  ↓ verification fails
Failed
```

## Payment Flow

### Customer Journey

```
Checkout
  ↓
Place Order
  ↓
Order Created (status: pending, payment_status: pending)
  ↓
Click "Proceed to Payment"
  ↓
POST /payments/initiate
  ↓
PaymentTransaction created (status: initiated → pending)
  ↓
Redirect to Flutterwave payment_link
  ↓
Customer completes payment on Flutterwave
  ↓
Flutterwave redirects to /checkout/return?status=...&tx_ref=...&transaction_id=...
  ↓
Frontend calls /payments/{reference}/verify
  ↓
Backend verifies with Flutterwave
  ↓
Order updated (status: paid, payment_status: paid)
  ↓
Stock decremented idempotently
  ↓
Customer sees payment success
```

### Webhook Journey

```
Flutterwave event
  ↓
POST /payments/callback
  ↓
PaymentCallbackRequest validates verif-hash signature
  ↓
PaymentService::handleCallback()
  ↓
If status is successful/completed → verifyAndProcess()
  ↓
Order and transaction updated
```

## Idempotency

- `verifyAndProcess()` locks the transaction row and short-circuits if the transaction is already `success`.
- Stock is decremented only when `stock_decremented` is false.
- Duplicate callbacks or verify calls do not create duplicate payments or stock movements.

## Failure Handling

| Scenario | Behaviour |
|----------|-----------|
| Initiation fails | Transaction status set to `failed`; customer sees error toast |
| Customer cancels on Flutterwave | Redirect `status=cancelled`; frontend shows retry options |
| Verification fails | Transaction status set to `failed`; order remains `pending` |
| Amount mismatch | Transaction marked `failed`; order not updated |
| Webhook signature invalid | Request rejected with 403; event logged |
