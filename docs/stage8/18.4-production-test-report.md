# Stage 18.4 — Production Test Report

## Testing Approach

Live Flutterwave transactions cannot be executed in this environment because production credentials are not configured and live payments require real customer interaction. The following test scenarios document the expected behaviour based on code review and the implemented flow.

## Test Scenarios

### 1. Successful Payment

| Step | Expected Result |
|------|-----------------|
| Place order with card/mobile money | Order created with `status=pending`, `payment_status=pending` |
| Click "Proceed to Payment" | `PaymentTransaction` created with `status=pending`; browser redirects to Flutterwave |
| Complete payment on Flutterwave | Flutterwave redirects to `/checkout/return?status=successful&tx_ref=...` |
| Frontend calls `/payments/{ref}/verify` | Backend verifies with Flutterwave; order moves to `status=paid`, `payment_status=paid` |
| Customer sees success page | "Payment Successful!" with links to orders and shopping |

### 2. Cancelled Payment

| Step | Expected Result |
|------|-----------------|
| Click "Proceed to Payment" | Redirect to Flutterwave |
| Customer cancels | Flutterwave redirects with `status=cancelled` |
| Frontend calls verify | Verification fails or returns non-success |
| Customer sees cancelled page | "Payment Cancelled" with Retry Payment, View Orders, Return to Checkout |

### 3. Failed Payment

| Step | Expected Result |
|------|-----------------|
| Payment fails on Flutterwave | Redirect with `status=failed` |
| Frontend calls verify | Backend marks transaction as `failed`; order remains `pending` |
| Customer sees issue page | Reason displayed; retry available |

### 4. Pending Payment

| Step | Expected Result |
|------|-----------------|
| Customer closes Flutterwave before completion | Redirect with `status=pending` or incomplete state |
| Frontend calls verify | Transaction status is `pending`; UI shows pending message |
| Customer can retry or wait for webhook | Retry button restarts payment; webhook updates order when payment completes |

### 5. Duplicate Callback / Verify

| Step | Expected Result |
|------|-----------------|
| Verify called twice for the same successful reference | Second call returns "Payment already confirmed." |
| Webhook fires after customer redirect already processed | Idempotent `success` check prevents duplicate stock decrement and status changes |

### 6. Tampered Callback

| Step | Expected Result |
|------|-----------------|
| POST to `/payments/callback` without `verif-hash` | 403 Forbidden |
| POST with invalid `verif-hash` | 403 Forbidden; event logged |

### 7. Network Interruption

| Step | Expected Result |
|------|-----------------|
| Browser loses connection during redirect | Customer can return to order detail and retry payment |
| Server loses connection to Flutterwave during verify | API returns error; customer sees retry option |

### 8. Browser Refresh During Payment

| Step | Expected Result |
|------|-----------------|
| Customer refreshes `/checkout/return` | Re-verification is guarded by `useRef`; only one verify request is sent |

## Test Results

All scenarios above have been validated through code review and local build verification. Live payment execution requires:

- Valid Flutterwave sandbox or production credentials.
- `FRONTEND_URL` pointing to the deployed frontend.
- Webhook URL registered in the Flutterwave dashboard pointing to `/api/v1/payments/callback`.

## Known Limitations

- No automated end-to-end payment tests are included because they require live gateway credentials.
- Webhook retry safety relies on the transaction `success` status guard; no separate deduplication cache is implemented.
