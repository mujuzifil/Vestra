# Stage 18.5 — Fulfilment Workflow Report

## Order Status Lifecycle

```
Pending
  ↓ Payment confirmed
Paid
  ↓ Admin action
Processing
  ↓ Admin action
Packed
  ↓ Admin action
Shipped
  ↓ Admin action
Delivered
```

Alternative paths:

```
Pending → Cancelled
Paid → Cancelled / Refunded
Processing → Cancelled / Refunded
Packed → Cancelled
Shipped → Delivered
Delivered → Refunded
```

## Transition Rules

Rules are enforced by `OrderStatusService::validTransitions`:

| Current Status | Allowed Next States |
|----------------|---------------------|
| pending | paid, cancelled |
| paid | processing, cancelled, refunded |
| processing | packed, cancelled, refunded |
| packed | shipped, cancelled |
| shipped | delivered |
| delivered | refunded |
| cancelled | (terminal) |
| refunded | (terminal) |

## Admin Actions

The Filament `OrderResource` provides:

| Action | Available From | Effect |
|--------|----------------|--------|
| Mark Paid | pending | Sets status and payment_status to paid |
| Mark Processing | paid | Starts fulfilment |
| Mark Packed | processing | Order packed |
| Mark Shipped | packed | Sets dispatched_at, sends notification |
| Mark Delivered | shipped | Sets delivered_at, sends notification |
| Cancel | pending/processing/packed | Cancels order, restores stock |
| Refund | paid/processing/delivered | Refunds order, restores stock |

## Audit History

Every transition creates an `OrderStatusHistory` record with:

- New status
- Notes
- Changed by user ID
- Timestamp

These records feed both the admin audit log and the customer timeline.

## Customer Visibility

- Order status is shown on order list, order detail, and tracking pages.
- Timeline events are built from `OrderStatusHistory` and displayed to the customer.
- Internal notes are never exposed to customers.
