# Stage 18.7 — Security Report

## Objective

Protect distributor data and operations through role-based access control, ownership policies, upload validation, credit authorization checks, and comprehensive audit logging.

## Role-Based Access & Middleware

Access to the distributor portal is gated by two layers:

1. `auth:sanctum` on all portal routes.
2. `EnsureDistributor` middleware, which checks `User::isDistributor()`.

`User::isDistributor()` requires both the `distributor` Spatie role and an active `Distributor` record. Users without the role or with a suspended account are rejected with `403 Forbidden`.

## Policies

Laravel policies enforce ownership at the model level:

- `DistributorPolicy` — view/update/manage only own distributor record or admin.
- `DistributorBranchPolicy` — view/update/delete only own branches or admin.
- `DistributorContactPolicy` — view/update/delete only own contacts or admin.
- `DistributorDocumentPolicy` — view/delete only own documents or admin; create requires distributor/admin role.

These policies prevent cross-account access. Automated tests confirm that one distributor cannot view another distributor's branches, contacts, or quotations.

## Document Upload Validation

Document and payment uploads are restricted by dedicated form requests:

- `StoreDocumentRequest` — file required, max 10 MB.
- `StorePaymentUploadRequest` — file required, `png/jpg/jpeg/pdf`, max 5 MB.

Files are stored on the `public` disk in scoped directories (`distributor-documents/`, `payment-uploads/`). The application does not execute uploaded files; they are served as static content only.

## Credit Authorization Checks

Credit checkout follows strict business rules:

- The `CreditAccount` must have `status = active`.
- The order total must not exceed `availableCredit()` (`limit - balance - authorized_amount`).
- Authorization is recorded as a transaction before the order is finalized.
- If authorization fails, checkout returns `422` with an error on `payment_method`.

This prevents distributors from exceeding their approved exposure.

## Audit Logging

Nearly every mutating action in the portal is logged via `AuditService`:

- Profile updates and logo changes
- Branch, contact, and document create/update/delete
- Quotation create/submit/accept
- Payment uploads
- Dashboard and statement views

Each log entry records the actor, event type, affected model, payload, IP address, and user agent.

## Additional Controls

- The public application form is rate-limited by `throttle:distributor`.
- Filament admin approval actions require confirmation.
- Sanctum provides CSRF protection for authenticated SPA requests.
