# Stage 18.7 — Branch Management Report

## Objective

Allow distributors to manage multiple operating branches, designate a default branch for orders, and keep delivery details current.

## Branch Data Model

Branches are stored in `distributor_branches` with the following fields:

- `distributor_id` (foreign key, cascade on delete)
- `name`, `manager_name`
- `phone`, `email`
- `country`, `district`, `city`, `address`
- `latitude`, `longitude`
- `delivery_notes`
- `status` (`active` / `inactive`)
- `is_default` boolean

The `DistributorBranch` model (`backend/app/Models/DistributorBranch.php`) provides `setAsDefault()`, which clears the default flag from all other branches belonging to the same distributor before marking the selected branch as default. The `Distributor` model exposes `defaultBranch()` to return the default branch or fall back to the first branch.

## Default Branch Handling

A default branch is seeded automatically during onboarding as `Head Office`, using the applicant's original address and contact details. Distributors can later change the default through the portal. The default branch is intended to be the primary delivery location for wholesale orders, although the checkout flow also supports selecting a different branch via `distributor_branch_id`.

## CRUD API

`App\Http\Controllers\Api\V1\Distributor\BranchController` exposes:

| Method | Endpoint | Description |
|--------|----------|-------------|
| GET | `/api/v1/distributor/branches` | List all branches for the distributor |
| POST | `/api/v1/distributor/branches` | Create a new branch |
| GET | `/api/v1/distributor/branches/{branch}` | View a specific branch |
| PUT | `/api/v1/distributor/branches/{branch}` | Update a branch |
| DELETE | `/api/v1/distributor/branches/{branch}` | Delete a branch |

Validation is handled by `StoreBranchRequest` and `UpdateBranchRequest`. Required fields are minimal (`name`); address and contact fields are optional. When `is_default` is sent as `true`, the controller calls `setAsDefault()` after save.

## Frontend

The branches page at `/distributor/branches` (`frontend/app/distributor/branches/branches-page-client.tsx`) provides:

- A grid of branch cards showing name, manager, contact, address, and default badge.
- Inline create/edit forms with fields for name, manager, contact, location, delivery notes, and default flag.
- Delete confirmation.
- Responsive layout that stacks forms and cards on mobile.

The `useDistributorBranches` React Query hook manages caching, mutations, and cache invalidation.

## Ownership Policies

`App\Policies\DistributorBranchPolicy` enforces ownership:

- `view` — allowed for the owning distributor or an admin.
- `update` — allowed for the owning distributor or an admin.
- `delete` — allowed for the owning distributor or an admin.

A distributor cannot view or modify another distributor's branches. Route model binding combined with the policy returns `403 Forbidden` for cross-account access attempts. This is covered by the automated test `test_distributor_cannot_view_other_distributor_branch`.

## Audit Logging

Create, update, and delete actions are logged via `AuditService` with the validated payload, IP address, and user agent.
